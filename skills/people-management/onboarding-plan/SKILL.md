# Onboarding Plan

Give a new hire a structured path to their first meaningful contribution — balancing learning (codebase, architecture, process) with doing (shipping something real, building relationships) — so every week has a clear "what good looks like".

## Inputs to gather

Gather these before drafting. If any are missing, ask in a single batched question — never invent the systems, the buddy, or the first project. Mark anything genuinely unavailable as **Unknown**.

- **New hire** — name, role, level
- **Team** — name and size
- **Key systems** — what they'll work on
- **Onboarding buddy** — who's paired with them
- **First project** — the planned first real project

## Steps

1. Write a **Pre-Day-One** checklist — access, accounts, dev environment, buddy briefed, welcome message, calendar of first-week 1:1s and walkthroughs, and a real (not toy) starter ticket prepared.
2. Plan **Week 1 (Orient and Connect)** — welcome, introductions, environment setup, architecture walkthrough, shadowing a review, first PR merged. End with a clear Day-5 success criterion.
3. Plan **Weeks 2-4 (First Real Contribution)** — 2-3 meaningful tickets independently, sprint ceremonies, key relationships (PM, QA, on-call), and starting the first project's design. End with a Day-30 success criterion.
4. Plan **Weeks 5-8 (Independent Delivery)** — build, test, review, ship the first project; shadow on-call. End with a Day-60 success criterion.
5. Plan **Weeks 9-12 (Full Contributor)** — second project, contributing to planning, writing/updating documentation, and a 90-day career conversation. End with a Day-90 success criterion.
6. Add **Ongoing Support** — weekly 1:1s (coaching-focused early), buddy pairing cadence, and structured 30/60/90-day check-ins in both directions.
7. Adapt to context: for a **senior hire (L5+)**, accelerate — leading a project by day 30, more autonomy, less hand-holding, but don't skip relationship-building. For a **junior hire (L3)**, extend the guided phase to 60 days with more pairing, smaller tickets, and a mentor (not just a buddy). For a **remote hire**, over-invest in social connection — virtual coffees with every team member, consider flying them in for week 1. For an **internal transfer**, skip company orientation but keep the full team/codebase onboarding.
8. Assemble the output in the format below.

## Boundaries

- Never invent the systems, buddy, first project, or team details. Mark unknowns as **Unknown** and ask.
- Keep success criteria observable and humane — a ramp plan, not a probation trap; don't set unrealistic week-one expectations.
- The starter ticket must be real and well-scoped, never a throwaway toy that signals busywork.
- Respect that ramp speed varies by person and background; the timeline is a default to adapt, not a verdict on the hire.

## Chaining

- After this, offer **one-on-one-prep** for the recurring 1:1s that carry onboarding through the first 90 days.
- At the 90-day mark, the career conversation can feed into **performance-review-draft** at the next cycle.
